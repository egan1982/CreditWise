## 0. 业务逻辑说明

### 0.1 解决什么问题？

风控建模中，坏样本（违约/欺诈）通常远少于好样本。例如 bad_rate=5% 意味着 100 个客户中只有 5 个"坏"的。这种**类别不平衡**会导致：

- **决策树**（规则挖掘）偏向预测多数类，难以发现少数类特征规则
- **逻辑回归**（评分卡）模型系数偏移，对坏样本的识别能力下降

之前系统的 AI 分析**能检测出不平衡问题并给出文字建议**，但没有实际的处理手段——用户看到建议也无法操作。

### 0.2 新增了什么？

一个参数 `imbalance_strategy`，提供 3 个选项：

| 选项 | 含义 | 适用场景 |
|------|------|---------|
| **自动选择**（默认） | bad_rate < 10% → 启用类别加权；≥ 10% → 不处理 | 大多数场景 |
| **不处理** | 什么都不做，维持原始行为 | bad_rate 本身较高（≥20%），或用户有特殊考量 |
| **类别加权** | 强制给少数类（坏样本）更高权重 | 用户明确知道需要加权 |

### 0.3 「类别加权」怎么工作？

不改变数据，不增减样本。只是告诉模型：**少数类的每个样本更"贵"**。

sklearn 的 `class_weight='balanced'` 会自动计算权重：

```
权重 = 总样本数 / (类别数 × 该类别样本数)

例：1000 样本，50 坏 950 好
  坏样本权重 = 1000 / (2 × 50)  = 10.0
  好样本权重 = 1000 / (2 × 950) = 0.526
```

这样模型在训练时会"重视"每一个坏样本，等效于把 50 个坏样本当成约 500 个来对待。

### 0.4 在两个任务中分别作用在哪？

```
规则挖掘任务：
  preprocessing ──→ generating_rules ──→ filtering ──→ selecting
                         ↑
                    决策树加 class_weight
                    （让树更关注坏样本分支）

评分卡开发任务：
  data_loading ──→ woe_binning ──→ model_training ──→ scoring
                       ↑                  ↑
                   ❌ 不受影响        逻辑回归加 class_weight
                  （WOE 必须用原始分布）
```

**关键约束**：评分卡的 WOE 分箱阶段**不能**用加权/采样数据，否则 WOE 值和 IV 值会失真。class_weight 只在模型训练阶段生效。

### 0.5 auto 策略的判定逻辑

```
bad_rate ≥ 20%  → 无不平衡，不处理
10% ~ 20%       → 轻度不平衡，不处理（class_weight 收益不大）
5% ~ 10%        → 中度不平衡，启用 class_weight
1% ~ 5%         → 重度不平衡，启用 class_weight
< 1%            → 极端不平衡，启用 class_weight

简化公式：bad_rate < 10% → class_weight，否则 → none
```

### 0.6 前端展示逻辑

第一阶段结果中新增**不平衡分析卡片**：

```
┌─────────────────────────────────────────────────┐
│ ⚖️ 类别不平衡分析                    [中度]      │
│                                                  │
│   1:18.5        类别加权        自动              │
│   好:坏比例     应用策略      用户选择            │
└─────────────────────────────────────────────────┘
```

颜色规则：
- **绿色** — 已启用处理（applied ≠ none）
- **琥珀色** — 存在不平衡但未处理，底部加 ⚠️ 建议
- **不显示** — bad_rate ≥ 20%（无不平衡，没必要展示）

### 0.7 AI 分析闭环

AI 分析收到 `imbalance_analysis` 数据后：
- 如果 severity=中度/重度 且 applied=none → **主动建议**：「建议调整'类别不平衡处理'参数为'自动选择'」
- 如果已启用 → 确认策略合理性

这样用户看到 AI 建议后，能直接在参数面板找到对应选项修改——形成**检测→建议→操作**的闭环。

#### MT-03 实测对比结果（2026-04-23）

**测试条件**：同一数据集 `starrel_train_with_amount.csv`（bad_rate≈5.13%），所有参数一致（mining_mode=multi, max_depth=5, n_vars=3, min_lift_filter=3.5, min_lift_ruleset=1.0, max_hit_rate_select=0.50），仅 `imbalance_strategy` 不同。

**规则集累积指标对比**：

| 指标 | A: none（不处理） | B: auto（class_weight） | 差异 |
|------|-----------------|----------------------|------|
| **最终规则数** | 31 条 | 21 条 | -32% |
| **累积命中率** | 14.06% | 12.06% | -2.0pp |
| **累积召回率** | 40.9% | 35.9% | -5.0pp |
| **累积坏账率** | 14.92% | 15.27% | +0.35pp |
| **累积 Lift** | 2.9 | 3.0 | +0.1 |
| **预估整体坏账率** | 3.53% | 3.74% | +0.21pp |
| **质量分** | 100.0 | 99.5 | -0.5 |
| **候选规则数** | 1678 | 1647 | -1.8% |
| **生成耗时** | 3.8s | 5.1s | +34% |

**业务解读**：

1. **none（宽策略）**：决策树按原始分布生长，多数类主导分裂方向。生成更多候选规则，筛选后保留31条。规则集覆盖更广（命中率14.06%、召回率40.9%），但 Lift 略低（2.9），"撒网更广但精度略低"。

2. **auto/class_weight（精策略）**：决策树给坏样本更高权重，分裂方向更偏向区分坏客户。单条规则质量更高（Top1 lift=9.7 vs 8.5），贪婪算法更早达到目标（21条满足 min_recall≥30%）。Lift 更高（3.0），"命中更准但覆盖略窄"。

3. **核心权衡**：

| 维度 | none（宽策略） | class_weight（精策略） |
|------|-------------|-------------------|
| 风控覆盖面 | ✅ 召回40.9% | 召回35.9% |
| 误伤率 | 命中率14% | ✅ 命中率12% |
| 精准度 | Lift 2.9 | ✅ Lift 3.0 |
| 运营复杂度 | 31条规则 | ✅ 21条更易管理 |

**结论**：差异符合预期，业务逻辑代码无误。如果业务侧更看重"不漏掉坏客户"选 none；更看重"精准拦截、少误伤"选 auto/class_weight。两者都是合理的风控策略选择。
