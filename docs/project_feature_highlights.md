## 一、一句话定位

**CreditWise 是一个 AI 驱动的信贷风控智能分析平台，将风控分析师最耗时、最依赖专业经验的"规则挖掘"和"评分卡建模"流程，从 2-5 天的人工操作缩短为分钟级自动完成。**

在信贷风控领域，最核心的两项工作——制定审批规则（什么样的客户应该拒绝）和开发评分卡（如何给客户信用打分）——长期依赖分析师手工编写 Python/SAS 代码完成。一个熟练的分析师完成一轮规则挖掘通常需要 3-5 天，评分卡建模需要 2-3 天，且过程难以复现和交接。CreditWise 通过将 AI 大模型的自然语言理解能力与确定性的 Pipeline 执行引擎深度结合，让业务分析师只需在浏览器中点选配置、用自然语言描述需求，即可在数分钟内获得专业级的分析结果和报告。

## 三、核心业务场景

### 场景一：新业务上线 — 从零搭建风控策略

> 🎯 **业务目标**：新产品上线前，需要制定一套贷前审批规则，在控制坏账率的前提下最大化通过率。

**典型背景**：某消费金融公司计划上线一款新的现金贷产品，历史没有该产品的风控数据，需要基于同类型产品的申请数据，在 3 天内制定出第一版审批规则交给决策引擎配置。

**传统方式的困境**：风控分析师需要打开 Jupyter Notebook，用 Python 逐特征探索——先写代码做数据清洗（处理缺失值、识别日期列和文本列），再对每个特征尝试不同的分箱方法和阈值组合，对生成的候选规则逐条计算召回率和命中率，最后手工挑选一组互不冲突的规则。这个过程通常需要 3-5 天，且每次调整参数（比如换一种分箱方法）都需要重新运行整个脚本。

**CreditWise 方式（3 步完成）**：

1. **上传数据**：将历史申请数据（CSV 或 Excel 格式）拖拽上传到平台，系统自动识别列类型、检测 ID 列和时间列
2. **配置参数**：在界面上指定目标变量（如 `is_default`），选择分箱方法（等频/等距/卡方/决策树）、设置规则数量上限和最低 Lift 阈值
3. **一键执行**：系统自动完成全流程：
   - 自动数据清洗（缺失值处理、特殊值替换、数据集划分）
   - 可选特征工程（日期衍生如 year/month/dayofweek、文本衍生如 length/word_count、分类变量 One-Hot 编码）
   - 决策树规则挖掘（支持单变量阈值和多变量组合两种模式）
   - 规则效果评估（计算每条规则的召回率、命中率、坏账率、Lift）
   - 规则方向检查（自动判断每个特征是"越小风险越高"还是"越大风险越高"）
   - 贪婪算法最优选择（在命中率上限约束下选出最优规则组合）
   - 规则质量自动检测（覆盖率、冲突、重叠、冗余四项检测 + 综合质量评分）
   - OOT 时间稳定性验证（计算规则在不同时间段样本上的 PSI 值）
   - 一键生成 Word/Excel/HTML 多格式分析报告

**实际产出**：一组可直接部署的审批规则（例如"年龄 ≤ 25 且收入 ≤ 5000 → 拒绝"），每条规则附带召回率、命中率、坏账率、Lift 等效果指标，加上 PSI 稳定性评级和质量检测报告，确保规则集不仅在历史数据上效果好、在未来也能保持稳定。

### 场景二：存量业务优化 — 评分卡模型迭代

> 🎯 **业务目标**：现有评分卡模型运行 6 个月后需要进行迭代更新，提升风险区分能力。

**典型背景**：某银行的信用卡申请评分卡已上线半年，随着宏观经济环境变化和客群结构变化，模型的 KS 值从上线初的 0.38 下降至 0.31。风控部门需要基于最近 6 个月的新样本数据重新开发评分卡。

**传统方式的困境**：建模工程师需要完整重走一遍建模流程——对每个特征手工试不同的分箱方法、检查 WOE 单调性、用 Python 做相关性分析和 VIF 检验、拟合逻辑回归、手工计算评分刻度参数（base_score、PDO），每一步都可能需要反复调整。至少 2-3 天才能完成一轮。

**CreditWise 方式**：

1. 导入近 6 个月的最新样本数据
2. 配置评分卡参数（base_score=600、PDO=50、分箱方法、IV 筛选阈值、是否启用逐步回归等）
3. 系统自动完成 7 阶段 Pipeline：
   - **数据加载与预处理**：自动划分训练集/测试集，处理缺失值
   - **WOE 分箱与 IV 分析**：对每个特征自动分箱，计算各分箱的 WOE 值和整体 IV 值，给出预测力评级（无/弱/中/强/极强），帮助分析师快速了解每个特征的区分能力
   - **多轮特征筛选**：依次进行 IV 筛选（粗筛，去掉预测力弱的特征）→ 相关性筛选（去冗余，两个高度相关的特征只保留 IV 更高的）→ VIF 检验（去共线性，确保入模变量互相独立）→ 逐步回归（可选，自动筛选最优特征组合）
   - **统计逻辑回归建模**：建模结果不仅包含系数，还自动输出每个变量的标准误、z 统计量、p 值、95% 置信区间，以及模型整体的 pseudo R²、AIC、BIC、似然比检验——让模型的统计显著性一目了然
   - **评分刻度转换**：将逻辑回归的概率映射为业务可用的信用评分（如 300-850 分），支持自定义 base_score 和 PDO 参数
   - **模型评估**：自动生成 KS 曲线、ROC 曲线、评分分布图、Lift 曲线，计算 KS/AUC/PSI/CSI 全套指标
   - **报告生成**：一键导出 Word（含图表嵌入）+ Excel（多 Sheet 数据明细）+ HTML（交互式图表）

**实际产出**：完整的评分卡表（每个变量的每个分箱对应多少分）+ 模型评估报告 + 新老模型 KS/AUC 对比 + 各特征稳定性分析，可直接提交模型评审委员会审核。

### 场景三：策略效果诊断 — 新老策略对比（规划中）

> 🚧 **状态**：方案设计中，预计 Q3 2026

当风控团队基于规则挖掘或评分卡开发产出新策略后，在上线前需要评估——用新策略替换旧策略，到底会带来什么变化？

**核心分析维度**：

- **Swap Set 四象限分析**：将客户群体按新旧策略的判断结果分为四类——新老都通过（一致通过）、新老都拒绝（一致拒绝）、老通过新拒绝（换出）、老拒绝新通过（换入），直观展示策略调整影响的范围
- **人数 + 金额双口径**：不仅看影响的客户数量，还要看涉及的金额规模——有些规则可能只影响少数客户但涉及大额贷款
- **业务影响链**：策略调整 → 通过率变化 → 放款量变化 → 收入变化
- **风险影响链**：策略调整 → 坏账率变化 → 预期损失变化 → 资产质量变化

这个功能将补全风控策略全生命周期的最后一环——让策略的上线决策有数据支撑。

## 四、两大核心 SOP 任务详解

### 4.1 规则挖掘 — 从数据到策略的自动化生产线

规则挖掘是信贷风控中最基础也最频繁的分析工作。CreditWise 将其标准化为 8 个阶段的自动化 Pipeline：

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 数据预处理│ → │ 特征工程  │ → │ 规则生成  │ → │ 规则筛选  │
│ ·自动清洗 │   │ ·日期衍生 │   │ ·决策树   │   │ ·Lift筛选│
│ ·质量评估 │   │ ·文本衍生 │   │ ·阈值分箱 │   │ ·命中率  │
│ ·数据划分 │   │ ·IV 筛选  │   │ ·方向检测 │   │ ·PSI稳定性│
└──────────┘   └──────────┘   └──────────┘   └──────────┘
                                                   ↓
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 报告生成  │ ← │ 金额分析  │ ← │ 先验分析  │ ← │ 最优选择  │
│ ·多格式   │   │ ·金额召回 │   │ ·增量贡献 │   │ ·贪婪算法 │
│ ·可导出   │   │ ·金额Lift │   │ ·重叠率   │   │ ·重叠模式 │
└──────────┘   └──────────┘   └──────────┘   └──────────┘
```

**各阶段业务价值详解**：

- **数据预处理**：自动识别并排除 ID 列、时间列、常量列等非建模字段；智能检测日期列和文本列；进行缺失值处理和数据质量评分（0-100 分）；支持三种数据集划分方式（手动标注 / 按时间 OOT / 随机划分）。这个阶段确保"垃圾进垃圾出"不会发生——分析师在后续阶段开始前就能看到数据质量报告。
- **特征工程（可选）**：从日期列自动提取 year/month/dayofweek/hour/days_since 等衍生特征；从文本列提取 length/word_count 等衍生特征；对分类变量自动 One-Hot 编码；计算各特征的 IV 值并按阈值筛选。这相当于把分析师过去需要手工写的 80 行特征工程代码全部自动化。
- **规则生成**：支持两种挖掘模式——单变量模式（对每个特征自动分箱生成阈值规则，如 `age ≤ 25`）和多变量模式（用决策树自动发现特征组合规则，如 `age ≤ 25 AND income ≤ 5000`）。系统会自动判断每个特征的风险方向（越小越高风险 vs 越大越高风险），排除方向不一致的规则。对于 One-Hot 编码的二值特征（如 `gender_is_male`），自动使用 `== 1` 方向。同时会自动检测数据泄露（完美分离特征）。
- **规则筛选**：计算每条候选规则的召回率（抓到了多少坏客户）、命中率（覆盖了多少客户）、坏账率（命中的客户中有多少是真的坏）、Lift（相对整体坏账率的提升倍数）；按最低 Lift 和最高命中率阈值过滤，并进行 PSI 稳定性预检。
- **最优规则选择**：支持两种选择算法——贪婪模式（每轮选出当前剩余样本中效果最好的规则，选完后移除已命中样本再继续选，确保规则间不重叠）和重叠模式（允许规则间命中相同客户，按坏账率排序依次选到命中率上限）；支持设置多目标约束（最低召回率、最低坏账率、目标坏账率）；可选 OOT 稳定性筛选（剔除在验证集上表现不稳定的规则）。
- **先验分析**：如果分析师有已有的业务规则，可以上传进行增量分析——看新规则相对于已有规则能多抓到多少坏客户、有什么样的互补性。
- **金额分析**：如果数据中有金额列（如贷款金额），可以在规则选择阶段加入金额维度——计算金额召回率和金额 Lift，确保选出的规则不仅在人数上有效、在金额规模上也有效。
- **报告生成**：自动生成包含累计指标曲线、规则质量验证报告、PSI 稳定性报告、决策树可视化、规则业务解读的完整分析报告。

**核心业务指标解读**：

| 指标 | 含义 | 业务判断标准 |
|------|------|------------|
| 召回率 (Recall) | 命中的坏客户占总坏客户的比例 | 越高越好——规则能抓到更多风险客户。但如果单条规则召回率过高（如 > 50%），可能过于宽泛 |
| 命中率 (Hit Rate) | 命中客户占总客户的比例 | 需要在抓坏和误伤间平衡。通常希望命中率在 5%-15% 之间——既能有效控制风险，又不会拒绝过多好客户 |
| 坏账率 (Bad Rate) | 命中客户中坏客户的占比 | 越高说明规则越精准。通常要求至少是整体坏账率的 2-3 倍以上 |
| Lift | 规则命中客户的坏账率 ÷ 整体坏账率 | 衡量规则的"提纯"效果。Lift > 2 才有实用价值，Lift > 5 说明规则非常精准 |
| PSI (Population Stability Index) | 规则命中率在不同时间段的稳定程度 | < 0.1 稳定可用；0.1-0.25 需关注；> 0.25 显著变化，规则可能已退化 |

**规则质量自动检测**：系统还会自动检测规则集的整体质量——覆盖率是否合理（太低抓不到风险，太高误伤太多）、规则间是否有逻辑冲突（如"年龄 ≤ 25 拒绝"和"年龄 ≤ 30 通过"同时存在）、重叠度是否过高（两条规则大量命中相同客户意味着冗余）、是否存在完全冗余的规则（规则 A 命中的客户完全被规则 B 覆盖），并给出综合质量评分（0-100 分）。

### 4.2 评分卡开发 — 端到端的信用评分建模

评分卡是信贷行业最核心的风险量化工具。CreditWise 将其标准化为 7 个阶段的自动化 Pipeline：

```
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ 数据加载  │ → │ WOE 分箱  │ → │ 特征选择  │ → │ 模型训练  │
│ ·预处理   │   │ ·IV 计算  │   │ ·相关性   │   │ ·逻辑回归 │
│ ·划分     │   │ ·单调性   │   │ ·VIF 共线 │   │ ·统计检验 │
└──────────┘   └──────────┘   └──────────┘   └──────────┘
                                                   ↓
┌──────────┐   ┌──────────┐   ┌──────────┐
│ 报告生成  │ ← │ 模型评估  │ ← │ 评分转换  │
│ ·多格式   │   │ ·KS/AUC  │   │ ·标准分   │
│ ·可导出   │   │ ·ROC/PSI │   │ ·概率映射 │
└──────────┘   └──────────┘   └──────────┘
```

**各阶段业务价值详解**：

- **数据加载与预处理**：自动划分训练集和测试集（支持随机划分和按时间 OOT 划分两种方式）；智能识别和排除非建模字段（ID 列、时间列、样本类型列）；缺失值统计和处理。这个阶段确保模型训练和验证的数据基础可靠。

- **WOE 分箱与 IV 分析**：这是评分卡开发最关键的步骤。系统对每个特征自动选择最优分箱方法（等频/等距/卡方/决策树），将连续值或分类值映射为 WOE（Weight of Evidence，证据权重）。WOE 值的大小和方向直接反映了该分箱对风险的影响——正值表示该分箱的坏客户比例低于整体水平（倾向于好客户），负值表示高于整体水平（倾向于坏客户）。理想的 WOE 值应随分箱呈单调趋势（要么递增、要么递减），系统会自动检查单调性并提示异常。IV（Information Value，信息值）则衡量整个特征对目标变量的预测能力——IV < 0.02 表示几乎无预测力，0.02-0.1 弱预测力，0.1-0.3 中等，0.3-0.5 强，> 0.5 极强但需警惕过拟合。

- **多轮特征筛选**：这是确保模型质量的关键步骤。系统按以下顺序依次筛选：① IV 筛选——直接剔除预测力太弱的特征（通常 IV < 0.02）；② 相关性筛选——两个高度相关的特征（如 Pearson 相关系数 > 0.7），只保留 IV 更高的那个，避免信息冗余；③ VIF 检验——计算方差膨胀因子（VIF > 5 或 10 表示存在严重多重共线性），迭代移除 VIF 最高的特征直到全部达标；④ 逐步回归（可选）——自动进行前向/后向/双向逐步回归，筛选出统计显著且贡献最大的特征组合。

- **统计逻辑回归建模**：使用逻辑回归（信用评分领域的标准建模方法），但不同于 sklearn 默认只输出系数，CreditWise 的 StatisticalLogisticRegression 会自动计算每个变量的完整统计信息——标准误（衡量系数估计的精确度）、z 统计量（系数是否显著不为零）、p 值（显著性水平，< 0.05 表示该变量对模型有显著贡献）、95% 置信区间（系数的可信范围）。模型整体还会输出 pseudo R²、对数似然值、AIC 和 BIC（用于模型比较和选择）、似然比检验（检验整个模型是否显著优于空模型）。对于好坏样本不均衡的数据，系统还支持类别不平衡处理（自动检测 bad_rate 并应用 class_weight 加权）。

- **评分刻度转换**：将逻辑回归输出的概率（0-1 之间，不直观）转换为业务人员熟悉的信用评分。系统支持自定义 base_score（基准分，如 600）、base_odds（基准 odds，如 1:20 即 20 个好客户对应 1 个坏客户）、PDO（Points to Double the Odds，odds 翻倍时分数的变化量，如 50）。转换公式确保评分的业务可解释性——评分越高代表风险越低，每增加 PDO 分，客户是好客户的 odds 翻倍。

- **模型评估**：自动生成全套评估指标和图表——KS 曲线（Kolmogorov-Smirnov，衡量模型区分好坏客户的最大能力，> 0.3 可用，> 0.4 良好，> 0.5 优秀）、ROC 曲线和 AUC 值（整体排序能力，> 0.7 可用，> 0.8 良好）、评分分布图（看不同分数段的好坏客户分布是否合理）、PSI（评分分布的稳定性，< 0.1 稳定）、CSI（各特征分箱在不同样本上的稳定性，< 0.1 稳定）。

- **报告生成**：一键导出 Word 报告（图表内嵌、格式专业）、Excel 报告（多 Sheet 数据明细，方便进一步分析）、HTML 报告（交互式图表，适合在线分享）。

**核心业务指标解读**：

| 指标 | 含义 | 业务判断标准 |
|------|------|------------|
| IV 值 | 单个特征对好坏客户的区分能力 | < 0.02 几乎无用；0.1-0.3 中等，值得入模；> 0.5 极强但需警惕过拟合 |
| KS 值 | 模型在最优切分点区分好坏客户的能力 | > 0.3 可用（有区分能力）；> 0.4 良好（区分明显）；> 0.5 优秀（区分很强） |
| AUC | 模型在所有可能切分点的平均排序能力 | > 0.7 可用；> 0.8 良好。AUC 不受切分点影响，比 KS 更稳定 |
| PSI | 评分分布在新老样本上的一致性 | < 0.1 稳定，模型可直接使用；0.1-0.25 需关注；> 0.25 模型可能已退化 |
| CSI | 每个特征分箱在新老样本上的稳定性 | < 0.1 稳定；> 0.1 需排查该特征的数据分布是否发生变化 |

**WOE/IV 分析示例**：假设数据中有一个"年龄"特征，系统会自动将其分箱（如 18-25 岁、26-35 岁、36-50 岁、50 岁以上），计算每个年龄段的 WOE 值（如 18-25 岁的 WOE = -0.5 表示该年龄段坏客户偏多，36-50 岁的 WOE = +0.3 表示该年龄段好客户偏多），并计算整体 IV 值（如 IV = 0.25，评级为"中等预测力"）。分析师查看这个结果后可以快速判断：年龄是一个有效的区分特征，年轻人风险更高，中年人风险更低，符合业务直觉。

## 五、差异化亮点

### 5.1 核心创新：LLM + Pipeline 混合架构 — "既要智能，又要可靠"

这是 CreditWise 最核心的架构创新：**将 AI 大模型的"智能理解"与确定性流程的"可靠执行"深度结合。**

**为什么不用纯 LLM？** 在项目探索阶段，我们尝试过让 AI 大模型完全主导整个分析流程（LLM SOP 模式），即 AI 在每一个分析阶段都自主决定下一步做什么、生成什么代码、如何解读结果。这个模式在概念验证阶段看起来很有吸引力——"让 AI 做一切"，但在实际使用中暴露了三个致命问题：

| 问题 | 具体表现 | 业务影响 |
|------|---------|---------|
| **AI 幻觉不可控** | LLM 每次可能选择不同的分析路径、生成不同的代码逻辑 | 同样的数据每次跑出不同的结果，风控策略的制定需要可复现性，这直接违背了金融行业的基本要求 |
| **调用成本高** | 一个完整的分析流程有 7-8 个阶段，每个阶段都要调用 LLM | 单次分析需要十几次甚至几十次 LLM 调用，Token 消耗大、响应延迟高 |
| **代码安全风险** | LLM 动态生成 Python 代码直接在服务器上执行 | 需要额外的沙箱隔离机制，且 LLM 生成的代码质量参差不齐，可能在处理边界情况时出错 |

**最终方案——分工明确**：

```
用户自然语言 ──→ LLM（"翻译官"）       ──→ Pipeline（"执行者"）
                  · 理解用户意图             · 预编译的标准化流程
                  · 提取任务参数             · 每一步结果可审计
                  · 不写任何代码             · 结果 100% 可复现
                  · 不参与执行               · 零 AI 幻觉风险

                  只在入口处介入一次！          全程确定性执行！
```

**这个架构带来的实际效益**：

- **成本降低约 80%**：LLM 只在任务开始时调用一次（参数提取），后续 7-8 个阶段全部是预编译代码执行，单次分析的 Token 消耗从数千降至数百
- **执行环节彻底消除 AI 幻觉**：Pipeline 的每个步骤都是经过测试和代码审查的确定性逻辑，不会出现"这次选了决策树、下次选了随机森林"的不一致情况
- **用户获得双重价值**：用自然语言交互的便利性（不需要学习参数含义），同时获得可复现、可审计的确定性结果（满足金融合规要求）
- **架构天然可扩展**：开发新的 SOP 任务时，只需要定义新的 Pipeline 流程和参数 Schema，完全不需要改动 LLM 交互层或执行引擎

### 5.2 "说人话就能做分析" — AI 智能入口

用户不需要了解任何参数的技术细节。在对话框中用自然语言描述需求即可：

> "帮我做规则挖掘，目标列是 is_default，分箱用决策树方法，最多选 5 条规则"
> "用最近 6 个月的数据更新评分卡，base_score 设 600，PDO 设 50"

系统会自动从自然语言中提取出结构化的任务参数——任务类型是什么（规则挖掘还是评分卡开发）、目标变量是哪个字段、分箱方法选哪种、各种阈值怎么设置——并自动启动对应的 Pipeline 执行。

更智能的是，如果用户的描述不完整（比如没指定分箱方法），系统会自动识别缺失的必需参数，并给出友好的提示："好的，要进行规则挖掘。请问分箱方法想用哪种？等频分箱（推荐）、等距分箱、卡方分箱还是决策树分箱？"用户在补充信息后即可继续。

**关键设计原则**：LLM 只做"翻译官"（将自然语言转为结构化参数），Pipeline 负责执行。这意味着每次你输入相同的需求，得到的是完全相同的结果——不会像纯 LLM 方案那样今天和明天给出不同的分析结论。

### 5.3 "边审边调" — 专家模式 + AI 建议 + 版本回溯

传统的全自动分析有一个问题：分析师对中间过程没有掌控感——要么全信系统的结果，要么完全手工重做。

CreditWise 的专家模式解决的就是这个问题。分析师可以在每个 Pipeline 阶段完成后**暂停审核**：
- 查看当前阶段的详细输出（如 WOE 分箱结果、特征筛选详情、候选规则列表）
- 如果满意 → 继续下一阶段
- 如果不满意 → 在界面上直接调整参数，然后**从当前阶段重新执行**（已完成的前序阶段会被缓存，不需要重新跑）

这意味着分析师可以把专业判断融入到自动化流程中——系统负责高效执行，人负责关键决策。这是"AI 自动化 + 人工审核"的最佳实践。

更值得一提的是 **AI 参数建议**机制。每次阶段完成后，系统会自动调用 LLM 分析当前阶段的输出，给出参数优化建议。例如，在 WOE 分箱完成后，AI 可能会分析："当前 max_depth=3，IV 值排名前 10 的特征中 3 个呈现非单调趋势，建议将 max_depth 调整为 5 以获得更精细的分箱"。分析师看到这个建议后，可以**一键采纳并自动重试**——不需要手动去改参数、重新配置。

每次重试都会自动保存**版本快照**，包括当时的参数、输出预览、AI 分析结果。分析师可以在界面上切换查看 v1、v2、v3 各个版本的结果，对比不同参数组合的效果差异。重试时 AI 还会自动读取上一版本的快照，在分析中注入版本对比信息："v1 版本的规则集 Lift 为 3.2，当前 v2 版本提升至 3.8，主要改进在于 max_depth 的增加使规则覆盖了更多交互特征。"

### 5.4 "不用写一行代码" — 从编程到配置的交互革命

传统风控分析的实际操作门槛远高于业务人员的技能范围：

| 传统分析任务 | 需要掌握的技能 | 典型代码量 | 门槛 |
|------------|--------------|-----------|------|
| 数据清洗与预处理 | pandas 数据处理、缺失值/异常值处理 | ~50 行 | 需要编程能力 |
| 特征工程 | 日期/文本衍生、One-Hot 编码 | ~80 行 | 需要编程能力 |
| 规则挖掘 | sklearn 决策树、阈值遍历、指标计算 | ~120 行 | 需要编程 + 算法知识 |
| 评分卡建模 | WOE/IV 理论、scorecardpy 库、逻辑回归 | ~150 行 | 需要统计学 + 编程 |
| 模型评估 | ROC/KS 计算、matplotlib 画图 | ~60 行 | 需要编程 + 可视化 |
| 报告生成 | Word/Excel 排版、图表嵌入 | ~50 行 | 费时且格式不统一 |
| **合计** | — | **~510 行代码** | **需要编程 + 统计学 + 风控领域知识** |

**CreditWise 将这一切从"写 500 多行代码"变成了"在浏览器里做 4 步操作"**：

```
传统方式（编程）                   CreditWise 方式（配置）
─────────────────────────          ─────────────────────
打开 Jupyter Notebook              打开浏览器
import pandas, sklearn...          拖拽上传数据文件
df = pd.read_csv(...)              ↓
# 写 50 行清洗代码                  下拉框选择目标变量
# 写 80 行特征工程代码              滑块/下拉框配置直观参数
# 写 120 行规则挖掘代码              ↓
# 写 150 行评分卡建模代码            点击"开始执行"
# 写 60 行评估画图代码              ↓
# 写 50 行报告导出代码              等待 2-5 分钟
                                   ↓
                                   下载专业报告（Word/Excel/HTML）
```

**核心转变**：

- **规则挖掘**：分析师只需指定"目标变量是谁、用什么分箱方法、最多选几条规则"，系统自动完成 8 个阶段的全部代码逻辑
- **评分卡开发**：分析师只需指定"base_score 多少、PDO 多少、分箱方法选哪个"，系统自动完成从 WOE 计算到模型评估的全部流程
- **参数优化**：不需要改代码重新跑，在界面上修改参数后直接从当前阶段重试，已完成的阶段结果自动缓存
- **报告导出**：一键生成三种格式，图表和表格自动排版，无需手动操作 matplotlib 或 Word

**这意味着什么？** 业务分析师不再需要会写 Python，不再需要理解决策树算法的细节，不再需要手动画 matplotlib 图表。**懂业务就够**——知道什么样的特征可能有用、什么样的规则组合是合理的、什么样的模型指标是达标的，剩下的逻辑实现全部交给系统。

### 5.5 "一套配置管所有渠道" — LLM 统一管理

在实际使用中，一个风控团队往往需要对接多个 LLM 供应商——可能是内部部署的 DeepSeek 做日常分析、Claude 做长文本报告、OpenAI 做代码生成。每个供应商有各自的 API Key、端点地址、模型参数。

CreditWise 内置的 LLM Manager 提供了统一的管理界面：
- 支持 OpenAI / DeepSeek / Claude / Google / 通义千问 / 企业内部 LLM 平台（如 Venus）六种供应商类型，以及任何兼容 OpenAI 格式的自定义 API
- 所有 API Key 采用 Fernet 加密存储（AES-128-CBC），前端展示时自动脱敏（只显示前后几位）
- 每个模型可以单独配置 temperature、top_p、max_tokens、system prompt，还支持联网搜索和深度思考等高级功能
- 负载均衡自动选择可用渠道——某个渠道不可用时自动切换到备选，确保服务不中断
- 完整的调用日志记录（请求内容、响应内容、Token 用量、响应时间），方便成本核算和使用审计

### 5.6 "数据不出网" — 内网私有化部署

金融行业对数据安全有极高的合规要求——客户申请数据、征信数据、还款记录等敏感信息不能离开企业内网。

CreditWise 支持完整的私有化部署：
- **Docker 一键部署**：一条命令完成所有服务（API、前端、LLM Manager、文件服务）的启动
- **Basic Auth 认证**：浏览器原生弹窗登录，支持 bcrypt 密码哈希存储，5 次失败自动锁定 15 分钟
- **角色权限**：管理员可以管理 LLM 渠道配置和用户账户，普通用户只能使用分析功能——防止 API Key 泄露和配置误改
- **数据全本地存储**：上传的数据文件、分析结果、LLM 调用日志、任务历史全部存储在本地磁盘，通过 Docker volume 持久化
- **仅出口流量**：只有 LLM API 调用需要访问外部网络（通过统一代理出口），数据文件从不出网

在内网环境中，系统还支持敏感信息预检——上传数据时自动进行双层检测（列名匹配 + 值扫描），识别可能包含身份证号、手机号、银行卡号等敏感字段的文件，高危内容直接阻断上传，中危内容给出警告后上传。账户有效期控制功能则允许管理员设置用户账户的过期时间，到期后自动禁止登录。

### 5.7 架构演进故事：从"让 AI 做一切"到"让 AI 做最擅长的事"

这个项目的架构不是一蹴而就的，而是经历了一次关键的技术路线转向：

```
Phase 1: 纯 LLM SOP 模式（探索期，约 2 周）
  ─────────────────────────────────────────────
  让 LLM 完全自主决定每个分析阶段的代码逻辑和执行路径
  → 发现的致命问题：
    · 每次分析结果不一致（AI 幻觉导致路径分叉）
    · LLM 调用成本过高（8 个阶段 × 每次调用）
    · 动态生成的代码质量不可控
  → 结论：纯 LLM 方案不适用于需要确定性的金融风控场景

Phase 2: LLM + Pipeline 混合架构（当前，稳定运行）
  ─────────────────────────────────────────────
  只让 LLM 做"入口翻译"（自然语言 → 结构化参数）
  Pipeline 做"执行"（预编译的确定性流程）
  → 获得的效果：
    · 结果 100% 可复现（相同参数 → 相同结果）
    · LLM 调用成本降低约 80%
    · 代码安全可控（预编译 + 代码审查 + 测试覆盖）
    · 兼顾了智能交互和可靠执行

Phase 3: 更多 SOP 任务扩展（规划中，架构已就绪）
  ─────────────────────────────────────────────
  策略诊断、风险预测、产品盈利试算、专家知识库接入……
  → 优势：架构已就绪，新任务只需定义 Pipeline 流程和参数 Schema
    不涉及 LLM 交互层或执行引擎的改动
```

这个演进过程本身就是一个有价值的经验：**不是 AI 能力不够强，而是要知道把 AI 放在什么位置才能发挥最大价值。** 在这个架构中，AI 负责它最擅长的（理解人类语言、提取结构化信息），确定性流程负责它最擅长的（可复现的计算和逻辑），两者各司其职、互不干扰。

## 八、下一步规划

| 方向 | 业务价值 | 预计时间 |
|------|---------|---------|
| **策略诊断 SOP** | 填补风控策略全生命周期的最后一环——策略上线前的效果评估。通过 Swap Set 四象限分析直观展示新旧策略的替换影响，用人次和金额双口径量化业务和风险两个维度的变化 | 2026 Q3 |
