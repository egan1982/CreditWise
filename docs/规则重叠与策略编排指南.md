## 1. 核心概念

### 1.1 规则生命周期

```
┌─────────────────────────────────────────────────────────────┐
│                      规则生命周期                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   【规则挖掘阶段】              【策略上线阶段】              │
│                                                             │
│   ┌─────────────┐              ┌─────────────┐             │
│   │ 是否允许重叠 │              │  规则漏斗    │             │
│   │ (选规则时)   │              │ (执行规则时) │             │
│   └─────────────┘              └─────────────┘             │
│         ↓                             ↓                     │
│   决定选哪些规则入库            决定规则怎么执行              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 关键术语

| 术语 | 定义 |
|------|------|
| **命中率** | 被规则命中的样本占总样本的比例 |
| **召回率** | 被规则命中的坏样本占总坏样本的比例 |
| **提升度** | 规则命中人群的坏账率 / 整体坏账率，衡量规则识别能力 |
| **规则重叠** | 同一样本被多条规则同时命中的情况 |
| **贪婪算法** | 逐步选择规则，每选一条就剔除已命中样本 |

## 2. 规则最优选择参数详解

### 2.1 最大命中率（规则集）

**定义**：所有选出来的规则加在一起，最多能"抓住"多少比例的客户。

**业务场景**：假设有10万客户，设置0.1（10%）意味着最终选出的规则集最多命中1万客户。

**为什么要限制？**
- 规则太"贪心"会误伤好客户
- 业务资源有限（如人工审核能力只够处理1万单）
- 留出空间给其他风控手段

---

### 2.2 最低召回率（规则集）

**定义**：选出的规则至少要"抓住"多少比例的坏人。

**业务场景**：如果历史坏账有1000人，设置0.3意味着规则集至少要能识别出300个坏人。

**适用场景**：
- 当你有明确的风控KPI（如"至少要识别30%的欺诈"）
- 当漏掉坏人的代价很高时

---

### 2.3 目标坏账率（规则集）⭐ 新增参数

**定义**：策略应用后期望达到的整体坏账率。

**核心理念**：**以终为始** - 你只需要告诉系统"我想把坏账率降到多少"，系统自动计算需要多少召回率。

**自动转换公式**：
```
召回率 = 1 - (目标坏账率 / 原始坏账率) × (1 - 最大命中率)
```

**业务场景**：
- 原始坏账率 5.13%，目标降到 3%，最大命中率 10%
- 系统自动计算：召回率 = 1 - (3% / 5.13%) × (1 - 10%) ≈ 47.4%

**使用建议**：
- 这是最直观的参数，直接设置你的业务目标
- 如果同时设置了"最低召回率"，系统会取较大值
- 目标坏账率不能高于原始坏账率（否则无需规则干预）

---

### 2.4 最低坏账率（规则集）

**定义**：被规则命中的这群人里，坏人占比至少要达到多少。

**业务场景**：确保规则够"准"，命中的人里至少有X%是坏人。

**示例**：设置最低5%，意味着规则抓住的客户群里，坏账率至少要5%。

> **注意**：原"最高坏账率"参数已废弃，由"目标坏账率"替代。

---

### 2.5 最低提升度（规则集）

**定义**：规则识别坏人的能力，要比"随便猜"强多少倍。

**解读**：
- 提升度 = 1 → 和随机抓没区别（没用）
- 提升度 = 3 → 比随机猜准3倍
- 提升度 = 10 → 比随机猜准10倍（很厉害）

**适用场景**：确保规则真的有"识别能力"，而不是碰运气。

---

### 2.6 允许规则重叠

**定义**：一个客户可以同时被多条规则命中吗？

| 开关状态 | 算法逻辑 | 适用场景 |
|---------|---------|---------|
| **关闭**（默认） | 贪婪算法，选了一条规则后，被它命中的客户就"剔除"，下一条规则只能从剩余客户里找 | 想让每条规则各管一块，不重复 |
| **开启** | 独立选择，规则之间可以有重叠命中 | 想最大化覆盖坏人，不在乎重复 |

---

### 2.7 PSI时间列

**定义**：用哪个时间字段来检测规则的稳定性。

**业务意义**：规则在不同时间段的表现是否一致。比如某规则在1月很准，2月就不行了，说明规则不稳定，不能上线。

---

### 2.8 参数速查表

| 参数 | 核心问题 |
|------|---------|
| 最大命中率 | 规则最多管多少人？ |
| 最低召回率 | 规则至少要抓住多少坏人？ |
| **目标坏账率** ⭐ | 策略应用后坏账率要降到多少？（自动算召回率） |
| 最低坏账率 | 被抓的人里坏人占比至少多高？ |
| 最低提升度 | 规则比瞎猜强多少倍？ |
| 允许重叠 | 一个人能被多条规则重复命中吗？ |
| PSI时间列 | 规则表现稳定吗？ |

### 2.9 规则淘汰原因说明

在最优选择阶段，未被选入最终规则集的规则会标注淘汰原因。不同模式下淘汰原因有所不同：

#### 通用淘汰原因

| 淘汰原因 | 说明 | 触发条件 |
|---------|------|---------|
| **坏账率为0** | 规则在样本集上无效 | 规则命中的样本中没有坏样本 |
| **命中率达上限** | 命中率配额已用完 | 累计命中率 ≥ 最大命中率参数 |
| **召回率目标已达成** | 已抓住足够多的坏人 | 设置了`min_recall_ruleset`且已达标 |
| **目标坏账率已达成** | 策略效果目标已满足 | 设置了`target_bad_rate_ruleset`且已达标 |

#### 允许重叠模式特有

| 淘汰原因 | 说明 |
|---------|------|
| **排序靠后（坏账率第X名）** | 按坏账率降序排列，排在已选规则之后 |

#### 贪婪模式特有

| 淘汰原因 | 说明 |
|---------|------|
| **样本被消耗（贪婪模式）** | 该规则原本命中的坏样本已被前面规则"吃掉"，剩余样本上坏账率降为0 |

#### 淘汰原因优先级

```
1. 坏账率为0（规则本身无效）
       ↓
2. 目标已达成（召回率/目标坏账率达标后截断）
       ↓
3. 命中率达上限（硬约束触发）
       ↓
4. 样本被消耗 / 排序靠后（模式特定）
```

## 3. 贪婪算法 vs 允许重叠

### 3.1 贪婪算法（不允许重叠）- 分工模式

**算法逻辑**：每条规则各管一块客群，互不干涉

```
规则A命中: 客户1、2、3
规则B命中: 客户4、5、6  ← 3被A命中了，B不能再要
规则C命中: 客户7、8     ← 每条规则负责独立的人群
```

**适用场景**：

| 场景 | 说明 |
|------|------|
| **差异化处置** | 不同规则命中后采取不同动作（规则A→人工审核，规则B→直接拒绝） |
| **规则归因清晰** | 出了问题能直接定位是哪条规则的锅 |
| **资源分配** | 每条规则消耗的审核资源可以单独统计和预算 |
| **策略迭代** | 下线某条规则，影响范围明确可控 |

**典型应用**：
- 信贷审批流程：不同规则对应不同的审批通道
- 反欺诈分层：高风险规则→拦截，中风险规则→人工，低风险规则→放行

---

### 3.2 允许重叠 - 协作模式

**算法逻辑**：多条规则可以同时命中同一个客户，形成"多重保险"

```
规则A命中: 客户1、2、3
规则B命中: 客户2、3、4  ← 客户2、3被两条规则同时命中
规则C命中: 客户3、5、6  ← 客户3被三条规则命中
```

**适用场景**：

| 场景 | 说明 |
|------|------|
| **最大化召回** | 只要被任意一条规则命中就拦截，确保坏人不漏网 |
| **打分叠加** | 多条规则命中的客户风险更高，可累加风险分 |
| **冗余设计** | 即使某条规则失效，其他规则还能兜底 |
| **简化决策** | 不care是哪条规则抓住的，只care有没有被抓住 |

**典型应用**：
- 黑名单机制：命中任意一条就拒绝
- 风险评分卡：每命中一条规则加X分，总分超阈值就拒绝
- 欺诈预警：多规则触发的客户重点关注

---

### 3.3 可视化对比

```
【不允许重叠】                    【允许重叠】
                                
┌─────┐ ┌─────┐ ┌─────┐        ┌─────────────────┐
│规则A│ │规则B│ │规则C│        │     规则A       │
│ 👤👤 │ │ 👤👤 │ │ 👤👤 │        │   👤  👤        │
└─────┘ └─────┘ └─────┘        │      ┌─────────┼───┐
   ↓        ↓        ↓          │      │  👤👤    │   │
 人工审核  直接拒绝  加灰名单    └──────┼─────────┘   │
                                       │    规则B    │
                                       └─────────────┘
                                              ↓
                                         统一处置
```

---

### 3.4 选择建议

| 你的业务需求 | 推荐设置 |
|-------------|---------|
| 不同规则要执行不同动作 | **不允许重叠** |
| 需要明确追溯是哪条规则拦的 | **不允许重叠** |
| 只要抓住坏人就行，不管用哪条规则 | **允许重叠** |
| 想用规则数量做风险打分 | **允许重叠** |
| 资源有限，每个客户只能处理一次 | **不允许重叠** |
| 宁可误伤也不能漏掉坏人 | **允许重叠** |

## 4. 策略编排：规则漏斗模式

### 4.1 什么是规则漏斗？

规则漏斗是策略执行时的**串行决策流程**：

```
客户进件
    ↓
┌─────────┐
│ 规则1   │ ──命中──→ 直接拒绝（出局）
└────┬────┘
     │未命中
     ↓
┌─────────┐
│ 规则2   │ ──命中──→ 人工审核（出局）
└────┬────┘
     │未命中
     ↓
┌─────────┐
│ 规则3   │ ──命中──→ 加灰名单（出局）
└────┬────┘
     │未命中
     ↓
  正常通过
```

### 4.2 漏斗模式特点

| 特点 | 说明 |
|------|------|
| **串行执行** | 按顺序一条条过 |
| **命中即止** | 一旦命中某条规则就出局，不再往下走 |
| **顺序敏感** | 规则1放前面还是后面，效果完全不同 |

### 4.3 漏斗 vs 并行执行

| 执行方式 | 逻辑 | 适用场景 |
|---------|------|---------|
| **漏斗（串行）** | 命中即止，按顺序处理 | 差异化处置、资源有限 |
| **并行** | 所有规则同时评估，汇总结果 | 风险打分、多重标记 |

## 5. 挖掘方式与执行方式的组合分析

### 5.1 四种组合方案

| 方案 | 挖掘阶段 | 执行阶段 | 效果 |
|------|---------|---------|------|
| A | 不允许重叠（贪婪） | 漏斗执行 | ✅ 天然契合 |
| B | 不允许重叠（贪婪） | 并行执行 | ⚠️ 可用，但并行意义不大 |
| C | 允许重叠 | 漏斗执行 | ⚠️ 后续规则效果"缩水" |
| D | 允许重叠 | 并行执行 | ✅ 天然契合 |

### 5.2 关键问题：方案C的效果是否等于方案A？

**答案：不一定一致**

#### 核心差异：选规则的"视角"不同

**方案A（贪婪不重叠）- 挖掘时就考虑顺序**

```
第1轮：从所有规则中选最优的 → 规则A（命中1,2,3,4,5）
       ↓ 剔除已命中样本
第2轮：从剩余样本中选最优的 → 规则B'（只看6,7,8...的表现）
       ↓ 剔除已命中样本  
第3轮：从剩余样本中选最优的 → 规则C'
```

**特点**：每条规则都是在"前面规则已经筛掉一波人"的前提下选的

**方案C（允许重叠挖掘 + 漏斗执行）- 挖掘时独立评估**

```
评估规则A → 在全量样本上表现很好 ✓
评估规则B → 在全量样本上表现很好 ✓
评估规则C → 在全量样本上表现很好 ✓
```

**特点**：每条规则都是在"全量样本"上评估的，没考虑彼此的遮挡

## 6. 实际案例分析

### 6.1 基础数据

假设有10000个客户，其中坏人1000个（坏账率10%）

| 规则 | 评估样本 | 命中人数 | 命中坏人 | 坏账率 | 提升度 |
|------|---------|---------|---------|-------|-------|
| 规则A | 全量 | 500 | 150 | 30% | 3.0 |
| 规则B | 全量 | 400 | 140 | 35% | 3.5 |
| 规则C | 全量 | 300 | 90 | 30% | 3.0 |

**关键信息：规则B和规则A有80%重叠！**（即规则B命中的400人中，有320人也被规则A命中）

---

### 6.2 方案C：允许重叠挖掘 → 漏斗执行

**挖掘阶段**：

根据全量表现选中：A、B、C（因为各自指标都不错）

预期效果：
- 规则A：命中500人，坏账率30%
- 规则B：命中400人，坏账率35%
- 规则C：命中300人，坏账率30%

**漏斗执行时（顺序A→B→C）**：

```
规则A命中500人（含150坏人）→ 出局

规则B本来命中400人，但其中320人已被A命中
→ 实际只新增80人（含28坏人，坏账率35%）

规则C本来命中300人，但部分已被命中
→ 实际新增150人（含45坏人）
```

**实际效果**：

| 规则 | 预期命中 | 实际新增命中 | 缩水比例 |
|------|---------|------------|---------|
| 规则A | 500 | 500 | 0% |
| 规则B | 400 | 80 | **80%** |
| 规则C | 300 | 150 | 50% |

**问题**：规则B在挖掘时看起来很厉害（提升度3.5），但实际贡献很小

---

### 6.3 方案A：贪婪不重叠挖掘

**挖掘阶段**：

```
第1轮：选规则A，命中500人（含150坏人）
       剔除这500人后，重新评估剩余规则
       
第2轮：规则B在剩余9500人中只命中80人
       但发现规则D在剩余样本中命中200人，坏账率40%
       → 选规则D（增量贡献更大）
       
第3轮：继续在剩余样本中找最优 → 选规则E
```

**最终选出**：A、D、E（而不是A、B、C）

**执行时**：每条规则的预期效果和实际效果一致

---

### 6.4 对比总结

| 对比维度 | 方案C（重叠挖掘+漏斗执行） | 方案A（贪婪不重叠挖掘） |
|---------|------------------------|---------------------|
| **规则选择依据** | 各规则在全量数据上的独立表现 | 各规则在"剩余样本"上的增量贡献 |
| **是否考虑规则间遮挡** | ❌ 挖掘时不考虑 | ✅ 挖掘时就考虑 |
| **选出的规则集** | 可能包含高度重叠的规则 | 倾向于选互补的规则 |
| **漏斗中后续规则效果** | 可能严重"缩水" | 保持预期效果 |
| **指标一致性** | 挖掘指标 ≠ 上线效果 | 挖掘指标 ≈ 上线效果 |

## 7. 最佳实践建议

### 7.1 决策流程图

```
                    你要怎么执行规则？
                          │
            ┌─────────────┴─────────────┐
            ↓                           ↓
       漏斗执行                      并行执行
       (串行，命中即止)              (打分/多重标记)
            │                           │
            ↓                           ↓
    推荐：贪婪算法挖掘             推荐：允许重叠挖掘
    (不允许重叠)                   
            │                           │
            ↓                           ↓
    ✅ 挖掘指标=上线效果           ✅ 充分利用规则信号
    ✅ 规则归因清晰                ✅ 多重保险
    ✅ 资源可控                    ✅ 风险分层打分
```

### 7.2 场景化建议

| 业务场景 | 推荐挖掘方式 | 推荐执行方式 |
|---------|------------|------------|
| 信贷审批流程（不同通道） | 贪婪（不重叠） | 漏斗 |
| 反欺诈拦截（宁杀错不放过） | 允许重叠 | 并行 |
| 风险评分卡 | 允许重叠 | 并行（累加分数） |
| 人工审核资源有限 | 贪婪（不重叠） | 漏斗 |
| 黑名单命中 | 允许重叠 | 并行（任一命中即拒） |
| 策略效果追溯 | 贪婪（不重叠） | 漏斗 |

### 7.3 避坑指南

| 坑 | 原因 | 解决方案 |
|----|------|---------|
| 允许重叠挖掘 + 漏斗执行，发现上线效果远不如预期 | 规则间重叠导致后续规则"缩水" | 改用贪婪算法挖掘 |
| 贪婪挖掘 + 并行执行，发现规则冗余度低 | 贪婪算法天然排斥重叠 | 如需并行执行，改用允许重叠挖掘 |
| 漏斗执行时改变规则顺序，效果大变 | 贪婪算法选规则时有隐含顺序 | 保持挖掘时的规则顺序 |

### 7.4 一句话总结

- **确定用漏斗执行** → 挖掘时用 **贪婪算法（不允许重叠）**
- **确定用并行执行** → 挖掘时用 **允许重叠**
- **不确定怎么执行** → 先用 **贪婪算法**，更稳妥
