## 一、部署与初始化

### 1.1 Docker 部署（推荐，Linux/CVM）

**前置条件**：已安装 Docker 和 Docker Compose。

**推荐方式：使用 `scripts/service.sh` 服务管理脚本**

```bash
# 启动（自动读取 .env 中的 ENABLE_AUTH 配置）
./scripts/service.sh start

# 其他常用命令
./scripts/service.sh stop        # 停止服务
./scripts/service.sh restart     # 重启服务
./scripts/service.sh status      # 查看运行状态 + 健康检查
./scripts/service.sh logs        # 实时跟踪日志（Ctrl+C 退出）
./scripts/service.sh build       # 重新构建镜像（代码更新后执行）
./scripts/service.sh hash <密码>  # 生成用户密码哈希
```

> 💡 `service.sh` 会自动从项目根目录的 `.env` 文件读取 `ENABLE_AUTH` 配置，无需每次手动传参。

**内网多用户模式**（启用 Basic Auth 认证）：

```bash
# 1. 准备用户配置文件
cp config/users.yaml.example config/users.yaml

# 2. 生成密码哈希
./scripts/service.sh hash <你的密码>

# 3. 编辑 config/users.yaml，填入生成的哈希值和用户信息

# 4. 在 .env 中启用认证
echo "ENABLE_AUTH=true" >> .env

# 5. 启动
./scripts/service.sh start
```

> 📸 *[截图占位：config/users.yaml 文件内容示例]*

**直接使用 docker-compose**（低级操作，适合调试）：

```bash
# 默认多用户模式（无需传 ENABLE_AUTH）
cd docker && docker compose up -d

# 单用户模式
ENABLE_AUTH=false docker compose up -d
```

> 默认部署为多用户模式，访问首页和 LLM Manager 均需登录。详见 [部署指南](./deployment_guide.md)。

启动后访问 `http://<服务器IP>:8200` 即可使用。如果启用了认证，浏览器会自动弹出登录框。

> 📸 *[截图占位：浏览器 Basic Auth 登录弹窗]*

### 1.2 非 Docker 部署（Windows 生产环境）

> ⚠️ **前置要求**：需安装 [PowerShell 7+ (pwsh)](https://github.com/PowerShell/PowerShell)。系统自带的 Windows PowerShell 5.1 **无法运行部署脚本**，因为 Git 仓库中的脚本文件使用 LF 换行，5.1 无法解析。

使用 `scripts/start_prod.ps1` 启动生产服务：

```powershell
# 默认端口 8200，监听 0.0.0.0（允许内网访问）
.\scripts\start_prod.ps1

# 自定义端口
.\scripts\start_prod.ps1 -Port 9000

# 停止服务
.\scripts\stop_prod.ps1
```

脚本会自动完成：端口占用检查 → 虚拟环境检测 → 设置 `DEV_MODE=false` → 启动后端 → 10 次健康检查。

日志写入 `logs/server.log`，可通过以下命令实时查看：

```powershell
Get-Content logs\server.log -Tail 50 -Wait
```

> ⚠️ **注意**：生产模式下（`DEV_MODE=false`），后端直接提供前端静态文件（`demo/chat/dist/`），**无需单独启动前端**。请确保在部署前已执行前端构建（`npm run build`）。

### 1.3 开发模式启动（本地调试）

```bash
# 安装依赖（首次）
pip install -r requirements.txt

# 启动后端（DEV_MODE=true，默认值）
python API/main.py
```

开发模式下前端由独立的 dev server 提供（端口 3001），需另行启动：

```bash
cd demo/chat
npm install  # 首次
npm run dev  # 启动前端 dev server（:3001）
```

> 📸 *[截图占位：python API/main.py 启动后的日志输出]*

### 1.4 部署验证

访问以下地址确认服务正常：

| 地址 | 说明 |
|------|------|
| `http://localhost:8200` | 主应用界面 |

## 二、LLM 渠道配置管理

### 2.1 进入 LLM Manager

在浏览器中访问 `http://<服务器IP>:8200/llm-manager`，进入 LLM 管理界面。

> 📸 *[截图占位：LLM Manager 主界面概览]*

### 2.2 创建 LLM 渠道

点击「新建配置」按钮，填写以下信息：

> 📸 *[截图占位：新建渠道配置表单]*

| 字段 | 说明 | 示例 |
|------|------|------|
| 配置名称 | 自定义标识，方便记忆 | "Venus-DeepSeek" / "生产环境 OpenAI" |
| 配置类型 | 选择供应商类型 | openai / deepseek / claude / google / qwen / openai_compatible |
| 模型名称 | 要使用的模型 ID | deepseek-chat / gpt-4o / claude-sonnet-4-20250514 |
| API Url | LLM 供应商的 API 端点 | https://api.deepseek.com/v1 |
| API 密钥 | 供应商提供的 API Key | sk-xxxxxxxx |

**类型选择指南**：

| 配置类型 | 适用场景 |
|---------|---------|
| openai | 使用 OpenAI 官方 API（GPT-4o、GPT-4.1 等） |
| deepseek | 使用 DeepSeek 官方 API，或任何 DeepSeek 兼容代理 |
| claude | 使用 Anthropic Claude API |
| google | 使用 Google Gemini API |
| qwen（通义千问） | 使用阿里云通义千问 API |
| openai_compatible（自定义） | 使用企业内部 LLM 平台（如 Venus），或其他兼容 OpenAI 格式的 API |

> 💡 **提示**：如果使用企业内部 LLM 平台（如 Venus），配置类型选择「自定义(openai_compatible)」，API Url 填写内部平台的代理地址，模型名称填写平台支持的模型 ID。

### 2.3 渠道测试

渠道创建或修改后，点击「测试连接」按钮验证：

> 📸 *[截图占位：渠道测试成功的结果展示]*

系统会通过代理发送测试请求，验证 API 密钥有效性和模型可用性。测试成功后，渠道状态自动设置为「激活」。

> ⚠️ **注意**：只有激活状态的渠道才会在分析任务中被使用。

### 2.4 模型参数高级配置

点击渠道列表中的「配置」按钮（或齿轮图标），进入模型参数配置弹窗：

> 📸 *[截图占位：模型配置弹窗界面]*

| 参数类别 | 参数 | 说明 | 默认值 | 建议范围 |
|---------|------|------|--------|---------|
| **基础** | temperature | 控制输出随机性，越低越确定 | 0.7 | 分析任务建议 0.1-0.4 |
| **基础** | top_p | 核采样阈值 | 1.0 | 0.8-1.0 |
| **基础** | max_tokens | 单次回复最大 Token 数 | 4096 | 2048-8192 |
| **高级** | frequency_penalty | 降低重复词频率 | 0 | -2 到 2 |
| **高级** | presence_penalty | 鼓励新话题 | 0 | -2 到 2 |
| **System Prompt** | system_prompt | 定义 AI 的角色和行为准则 | — | 见下方预设说明 |
| **功能** | enable_web_search | 启用联网搜索（仅 DeepSeek 等支持） | 关闭 | — |
| **功能** | enable_deep_thinking | 启用深度思考模式（仅 Claude 等支持） | 关闭 | — |

**预设 System Prompt 说明**：

| 预设 | 使用场景 | 推荐的 temperature |
|------|---------|:--:|
| 通用对话 | 日常问答、数据咨询、概念解释 | 0.7 |
| 参数推断 | 从自然语言提取 SOP 任务参数 | 0.1 |
| 结果解释 | 将分析结果转为业务语言 | 0.4 |

> 📸 *[截图占位：预设配置下拉选择效果]*

### 2.5 负载均衡

当多个同类型渠道同时激活时，系统自动进行负载均衡——请求在这些渠道间智能分发。如果某个渠道调用失败，会自动切换到下一个可用渠道。

> 💡 **最佳实践**：建议为重要任务（如评分卡开发）配置至少 2 个激活的渠道作为备份，确保分析任务不因单个渠道故障而中断。

### 2.6 调用日志

所有通过 LLM Manager 代理的 LLM 调用都会记录详细日志：

> 📸 *[截图占位：API 日志列表页面]*

每条日志包含：请求时间、模型名称、请求内容摘要、响应 Token 数、响应时间、成功/失败状态。管理员可通过日志了解 LLM 使用情况，估算成本。

## 三、用户与安全管理

### 3.1 用户管理（内网多用户模式）

用户账号通过 `config/users.yaml` 文件配置。管理员可编辑该文件添加、修改或删除用户。

> 📸 *[截图占位：config/users.yaml 完整内容示例]*

```yaml
users:
  - username: admin
    password_hash: "$2b$12$..."  # 使用 scripts/hash_password.py 生成
    role: admin
    enabled: true

  - username: analyst01
    password_hash: "$2b$12$..."
    role: user
    enabled: true
    expires_at: "2026-12-31"     # 可选：账户过期日期
```

**角色权限说明**：

| 权限 | admin（管理员） | user（普通用户） |
|------|:--:|:--:|
| 使用 AI 对话和分析任务 | ✅ | ✅ |
| 查看自己的任务历史 | ✅ | ✅ |
| 导出分析报告 | ✅ | ✅ |
| 访问 LLM Manager 管理界面 | ✅ | ❌ |
| 创建/编辑/删除 LLM 渠道 | ✅ | ❌ |
| 查看 LLM 调用日志 | ✅ | ❌ |
| 管理用户账户 | ✅ | ❌ |

> 📸 *[截图占位：管理员视角 vs 普通用户视角的界面差异对比]*

### 3.2 生成密码哈希

```bash
# 方式一：使用 service.sh（推荐，服务已启动时）
./scripts/service.sh hash <你的密码>

# 方式二：使用 docker run（离线部署场景，服务未启动时）
docker run --rm creditwise:latest python scripts/hash_password.py <你的密码>

# 方式三：在已运行的容器内执行
docker-compose exec creditwise python scripts/hash_password.py <你的密码>
```

将输出的哈希值填入 `config/users.yaml` 中对应用户的 `password_hash` 字段。

### 3.3 敏感信息检测

系统在上传数据文件时会自动进行敏感信息检测：

- **L1 列名检测**：检查列名是否匹配身份证号、手机号、银行卡号、姓名等敏感字段模式
- **L2 值扫描**（可选）：对列内容进行正则匹配，检测是否实际包含敏感数据

> 📸 *[截图占位：敏感信息检测告警弹窗]*

- 高危命中：直接阻断上传，提示用户脱敏后重新上传
- 中危命中：弹出警告弹窗，用户确认后仍可上传

## 四、界面总览

CreditWise 采用**三列式布局**，左列为导航控制区，中列为主内容与对话区，右列为详情展示区：

> 📸 *[截图占位：三列式界面全景截图]*

```
┌──────────────────┬──────────────────────┬──────────────────────────┐
│   左列（导航控制） │    中列（主内容）      │     右列（详情展示）       │
│                  │                      │                          │
│  · 文件上传与管理  │  · AI 对话交互        │  · 代码编辑器             │
│  · TaskType 选择  │  · 任务执行进度       │  · 阶段输出预览           │
│  · 交互模式切换   │  · SOP 任务结果      │  · 结果可视化             │
│  · 历史任务管理   │  · 模型选择器（底部）  │  · AI 分析建议            │
│                  │                      │                          │
└──────────────────┴──────────────────────┴──────────────────────────┘
```

| 区域 | 主要用途 | 什么时候用 |
|------|---------|-----------|
| **左列（导航控制）** | 管理文件、选择任务类型、切换交互模式、查看历史任务 | 上传数据、选择规则挖掘/评分卡、切换专家模式 |
| **中列（主内容）** | 与 AI 对话、执行任务、查看阶段输出、审核结果 | 对话交互、执行 Pipeline、查看各阶段产出、切换模型 |
| **右列（详情展示）** | 查看代码、可视化结果、AI 分析建议 | 查看代码实现、接受 AI 参数建议 |

## 六、规则挖掘任务

### 6.1 启动规则挖掘

有两种方式启动规则挖掘任务：

**方式一：自然语言启动（推荐）**

在中列对话区输入描述，系统自动识别意图并提取参数：

> 💬 "帮我做规则挖掘，目标变量是 is_default，分箱用决策树方法，最多选 5 条规则"

> 📸 *[截图占位：对话中输入自然语言指令的效果]*

系统识别到规则挖掘意图后，会提取参数并在中列弹出任务确认卡片，展示识别到的参数供你确认或修改。

> 📸 *[截图占位：TaskConfirmCard 任务确认卡片]*

**方式二：手动配置启动**

在中间列的任务配置面板中，选择任务类型为「规则挖掘」，然后手动填写各项参数：

> 📸 *[截图占位：规则挖掘任务参数配置面板]*

### 6.2 规则挖掘参数说明

| 参数类别 | 参数 | 说明 | 推荐值 |
|---------|------|------|--------|
| **基础** | 目标变量 (target_col) | 标识好坏客户的列名，如 is_default | 必填 |
| **基础** | 权重列 (weight_col) | 样本权重列（可选，无则等权） | 可选 |
| **挖掘方式** | mining_mode | single=单变量规则 / multi=多变量组合规则 | multi |
| **单变量** | n_bins | 分箱数量 | 10 |
| **单变量** | bin_method | 分箱方法：quantile(等频) / uniform(等距) / chi2(卡方) / tree(决策树) | tree |
| **单变量** | rule_directions | 规则方向：both(双向) / <= / > | both |
| **多变量** | n_vars | 每条规则最多包含几个特征 | 3 |
| **多变量** | max_depth | 决策树最大深度 | 5 |
| **多变量** | min_samples_leaf | 叶节点最小样本占比 | 0.01（1%） |
| **多变量** | use_full_tree | 是否使用全特征树（true=一期树提取，false=特征组合遍历） | false |
| **单个规则筛选** | min_lift_filter | 最低 Lift 阈值 | 3.5 |
| **单个规则筛选** | max_hit_rate_filter | 最高命中率阈值 | 0.03（3%） |
| **规则集选择** | max_hit_rate_select | 规则集总体命中率上限 | 0.20（20%） |
| **规则集选择** | allow_overlap | 是否允许规则重叠（true=重叠模式，false=贪婪算法） | false |
| **类别不平衡** | imbalance_strategy | 好坏样本不均衡处理策略 | auto |
| **特征工程** | enable_feature_engineering | 是否启用特征工程阶段 | false（关闭可加速） |
| **OOT 验证** | enable_oot_validation | 是否启用 OOT 验证 | false |
| **OOT 验证** | oot_ratio | OOT 验证集占比（需指定 time_col） | 0.1（10%） |

> 📸 *[截图占位：参数面板中各项参数的 Tooltip 帮助提示]*

### 6.3 执行与进度查看

点击「开始执行」后，系统会按 8 个阶段依次执行。每个阶段完成后，中列会更新阶段状态卡片：

> 📸 *[截图占位：Pipeline 执行过程中各阶段状态卡片的动态变化]*

阶段状态标识：
- 🔵 执行中 —— 当前正在运行的阶段
- ⚪ 待执行 —— 还未开始的阶段
- ✅ 已完成 —— 执行成功的阶段
- ❌ 失败 —— 执行出错的阶段

## 八、专家模式与任务控制

### 8.1 什么是专家模式

专家模式允许你在每个 Pipeline 阶段完成后暂停审核，而不是让系统全自动执行到底。

**启用方式**：在启动任务时，将交互模式选择为「专家模式」（expert）。

> 📸 *[截图占位：交互模式选择开关「全自动」/「专家模式」]*

### 8.2 专家模式下的操作

每个阶段完成后，系统自动暂停，你需要查看当前阶段的输出并做出决策：

> 📸 *[截图占位：专家模式暂停状态下的界面——阶段结果 + 操作按钮]*

- ✅ **继续**：当前阶段结果满意，继续执行下一阶段
- 🔄 **重试**：对结果不满意，修改参数后从当前阶段重新执行
- ⏸️ **保持暂停**：暂时不做决定，稍后再处理

### 8.3 AI 分析建议

在专家模式下，每个阶段完成后系统会自动调用 AI 分析当前阶段的输出，生成优化建议：

> 📸 *[截图占位：AI 分析建议卡片（SuggestedParamsCard），展示参数 diff 和两个操作按钮]*

AI 建议卡片会显示：
- 💡 当前参数值（灰色删除线）→ 建议参数值（绿色）
- 两个操作按钮：
  - 「仅填入参数」：将建议填入参数面板，不自动重试
  - 「填入并立即重试」：填入参数后立即从当前阶段重新执行

> 💡 **最佳实践**：AI 建议是基于数据统计给出的参考，不一定每次都采纳。建议先查看分析文本中的推理过程，再决定是否使用建议的值。

### 8.4 版本快照与回溯

每次阶段重试都会自动保存一个**版本快照**，包含当时的参数、阶段输出和 AI 分析结果。

> 📸 *[截图占位：StageVersionSelector 版本下拉选择器]*

在阶段输出预览区域的右上角，有一个版本选择下拉框，可以切换查看不同版本的输出结果。当前版本显示为「当前」，历史版本显示为「v1 / v2 / v3 + 时间 + 重试原因」。

选择历史版本后，页面会展示该版本的完整输出，并显示蓝色信息条：「📋 这是 v1 版本的结果（2026-06-12 15:30 · 调整 max_depth）。点击此链接用该版本参数重新执行当前阶段。」

### 8.5 任务暂停、停止与恢复

在任务执行过程中，可以随时进行控制：

> 📸 *[截图占位：任务执行中的控制按钮——暂停/停止]*

| 操作 | 效果 | 适用场景 |
|------|------|---------|
| ⏸️ **暂停** | 当前阶段完成后停止，状态变为 paused | 需要暂时离开、想先看当前阶段结果 |
| ⏹️ **停止** | 立即停止执行，状态变为 stopped | 发现参数配错、数据有问题需要重新来 |
| ▶️ **恢复** | 从暂停点继续执行 | 暂停后确认无误，继续后续阶段 |

> ⚠️ 停止后不可恢复，需要重新创建任务。暂停状态下可以随时恢复。

暂停的任务状态会持久化到磁盘，即使服务器重启也不会丢失。

## 附录：常见问题

**Q：为什么启动任务后没有反应？**

A：请确认已在 LLM Manager 中激活了至少一个 LLM 渠道。如果没有激活的渠道，任务中的 AI 分析功能将无法工作。

**Q：上传的 CSV 文件中文乱码怎么办？**

A：请确保 CSV 文件使用 UTF-8 编码保存。在 Excel 中另存为 CSV 时，选择「CSV UTF-8（逗号分隔）」格式。

**Q：规则挖掘结果中某条规则的命中率为 0？**

A：可能是规则方向配置有误。请检查该特征的风险方向——使用「专家模式」查看特征方向检测结果，或在参数中手动指定方向。

**Q：评分卡的 KS 值偏低（< 0.3）怎么办？**

A：从 WOE/IV 分析入手——检查是否有特征的 WOE 值非单调，尝试更换分箱方法（如从等频改为决策树分箱）；或在特征筛选中降低 IV 阈值和 VIF 阈值，让更多有效特征入模。

**Q：如何在不同 LLM 供应商之间切换？**

A：在顶部模型选择器中直接选择即可，不需要重启任务。不同任务、不同阶段可以使用不同的模型。

**Q：数据文件太大导致上传失败？**

A：建议先对数据进行抽样（如取最近 3 个月），或在本地做初步特征筛选后再上传。上传限制为 200MB。
